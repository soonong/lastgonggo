# 수집 룰

## 기본 원칙

수집 원본은 `서버공고(A1)`에서 받은 전체 컬럼 데이터다.
발주처 홈페이지, A3 파싱용공고문, A4 적격심사세부기준은 수집 원본이 아니라 전처리 중 참조하는 보조 데이터다.

## 기록 항목

| 항목 | 설명 | 예시 |
|---|---|---|
| 발주처 | 서버공고 row 안의 발주처 컬럼 | 한국전력공사 |
| 업무유형 | 서버공고 row 안의 시설, 용역, 물품 등 구분 | 시설 |
| 원본 데이터 | 서버공고 API 전체 row | 원본 JSON |
| 첨부파일 | 전처리 참조용 공고문 파일 목록 | 공고문 PDF |
| 수집상태 | 성공, 실패, 일부실패 | 첨부파일 다운로드 실패 |

## bidding2.kr 수집 기준

상세 조사 내용은 `docs/10_API_엔드포인트_조사.md`를 따른다.

수집 기준:

```text
1. 서버공고: bidDataOrigin_get.php?moduleKey=...&isDefault=Y&containCancel=N&onlyGong=Y
```

예시:

```text
서버공고에서 `2601767-00` row의 모든 컬럼을 받는다.
이 row가 수집 원본이다.
```

전처리 참조 API:

```text
A2 공고문첨부파일: 공고문 파일 목록 확인
A3 파싱용공고문: 문서곡괭이 본문 파싱
A4 적격심사세부기준: 적격 기준 row 조회
A5 웹용공고문: 사람이 상세 화면 확인
```

주의:

```text
bidDataOrigin_get.php는 `moduleKey`가 없으면 `권한이 없습니다.`를 반환한다.
브라우저에서는 직접 외부 URL을 호출하지 않고 로컬 proxy에서 `moduleKey`를 주입한다.
qualification.php는 `gongsanum`으로 조회하지 않는다.
`적격기준세부`, `건설` 파라미터를 사용하고, 브라우저에서는 `/api/proxy`를 경유한다.
```

## API 이름 기준

| ID | 이름 | 사용 기준 |
|---|---|---|
| A1 | 서버공고 | 서버에서 내려받는 공고 원본 목록 |
| A2 | 공고문첨부파일 | 전처리 참조용 파일 목록 |
| A3 | 파싱용공고문 | 문서곡괭이용 HTML 본문 |
| A4 | 적격심사세부기준 | 적격심사 기준 row 목록 |
| A5 | 웹용공고문 | 사람이 웹에서 확인하는 bidding2.kr 상세 화면 |

처리 예시:

```text
빈 `참가자격`을 채우려면 A3 파싱용공고문을 먼저 본다.
사용자가 원문 화면을 확인해야 하면 A5 웹용공고문 링크를 연다.
```
