# 설정 마스터 룰

## 목적

전처리 기준이 되는 파일은 코드에 고정하지 않고 웹 설정 화면에서 관리한다.
사용자는 설정 화면에서 기준 파일을 추가, 변경, 수정할 수 있어야 한다.

예시:

```text
적격발주처 매핑 파일을 수정하면, 다음 전처리부터 변경된 매핑 기준이 적용된다.
```

## 적격발주처 매핑

첨부 파일:

```text
C:/Users/imbid/Downloads/적격발주처_변경_2026-05-22.csv
```

파일 구조:

| 컬럼 | 의미 | 예시 |
|---|---|---|
| id | 표시/정렬용 번호 | `1` |
| 발주처 키워드 | 원본 발주처에서 찾을 키워드 | `한국남부발전` |
| 매핑 대상 | 키워드가 매칭되면 사용할 표준 발주처 | `조달청` |

현재 행:

| id | 발주처 키워드 | 매핑 대상 |
|---|---|---|
| 1 | 한국농어촌공사 | 한국농어촌공사 |
| 2 | 한국수력원자력 | 한국수력원자력 |
| 3 | 한국전력공사 | 한국전력공사 |
| 4 | 한국남부발전 | 조달청 |
| 5 | 한국동서발전 | 조달청 |
| 6 | 한국남동발전 | 조달청 |
| 7 | 한국서부발전 | 조달청 |
| 8 | 한국중부발전 | 조달청 |

처리 방향:

```text
원본 발주처 또는 적격발주처 관련 문자열에 `발주처 키워드`가 포함되면 `매핑 대상`을 `원발주처`로 사용한다.
키워드 매칭은 포함검색으로 처리한다.
```

예시:

```text
원본 발주처=`한국남부발전 주식회사`
발주처 키워드=`한국남부발전`
매핑 대상=`조달청`
  -> 원발주처=`조달청`
```

주의:

```text
`적격발주처`는 서버공고에서 값이 올 수 있지만 항상 채워지는 값으로 보지 않는다.
값이 있으면 수집 출처와 홈페이지 연결 기준 후보로 사용한다.
`원발주처`는 실제 평가에 사용할 적격심사기준 발주처다.
따라서 이 파일의 `매핑 대상`은 `적격발주처`를 덮지 않고 `원발주처`를 채운다.
```

확정 규칙:

| 항목 | 규칙 | 예시 |
|---|---|---|
| 적용 컬럼 | `매핑 대상`은 `원발주처`에 넣는다 | `한국남부발전 -> 조달청`이면 `원발주처=조달청` |
| 매칭 방식 | 포함검색 | `한국남부발전 주식회사`에 `한국남부발전`이 포함되면 매칭 |
| 적격발주처 보존 | `적격발주처` 값이 있으면 홈페이지 연결 기준 후보로 보존하고, 빈값이면 억지로 채우지 않는다 | `적격발주처=한국남부발전`이면 보존 |

## 설정 화면 요구사항

적격발주처 매핑은 웹 설정에서 편집할 수 있어야 한다.

필수 기능:

```text
행 추가
행 수정
행 삭제
CSV 가져오기
CSV 내보내기
사용 여부 관리
```

권장 컬럼:

| 컬럼 | 필요성 | 예시 |
|---|---|---|
| id | 정렬과 관리 | `4` |
| 발주처 키워드 | 매칭 조건 | `한국남부발전` |
| 매핑 대상 | 표준값 | `조달청` |
| 사용여부 | 임시 비활성화 | `true` |
| 메모 | 사람이 보는 설명 | `발전사는 조달청 기준 적용` |

예시:

```text
`한국남부발전 -> 조달청` 규칙을 잠시 쓰지 않으려면 삭제하지 않고 `사용여부=false`로 둔다.
```

## 지역설정 마스터

지역 관련 기준은 같은 `지역설정` 화면에서 관리하되, 성격에 따라 테이블을 분리한다.

권장 탭:

```text
지역설정
  - 행정지역
  - 배전사업소
  - 송전사업소
  - 폐광지역진흥지구
```

이유:

```text
행정지역은 지역제한, 검색용현장, 일반 공사현장 표준화에 쓰인다.
배전사업소와 송전사업소는 한전 사업소 기준이며, 폐광지역진흥지구는 특수 지역 기준이다.
같은 화면에서 관리하되 테이블을 섞으면 컬럼 의미가 달라져 유지보수가 어려워진다.
```

필수 기능:

```text
행 추가
행 수정
행 삭제
CSV 가져오기
CSV 내보내기
사용 여부 관리
```

예시:

```text
배전사업소에서 `강남지사`의 인접 사업소가 바뀌면
지역설정 > 배전사업소 탭에서 `인접 사업소`만 수정한다.
```

## 적격심사기준 관리 구조

적격심사기준 관련 마스터는 기존 참고 파일명을 그대로 고정하지 않고,
우리 시스템 기준으로 다시 구조화한다.
세부 구조는 `docs/17_적격심사기준_분류구조_초안.md`를 기준으로 검토한다.

권장 그룹:

```text
룰관리
  - 적격심사기준
    - 기준대상/제외 입찰방식
    - 발주처 정규화
    - 1차 기준표
    - 2차 세부조건
    - 미등록 발주처
```

참고자료의 파일명은 구현 후보로만 본다.

| 참고명 | 우리 구조에서의 위치 | 예시 |
|---|---|---|
| `rules_적격_skipKeywords.json` | 기준대상/제외 입찰방식 | `수의계약`이면 A4 호출 제외 |
| `적격발주처 변경` | 발주처 정규화 | `한국남부발전 -> 조달청` |
| `master_evaluationCriteria.json` | 1차 기준표 | `한국전력공사 + 일반건설 + 3억미만` |
| `orgCriteria` | 2차 세부조건 | `공고확인=배전공가`이면 배전공가 기준으로 덮기 |

2차 세부조건은 첨부된 `적격심사기준_변경_2026-05-24.csv` 기준으로
A~H 조건 영역, I~K 핵심 덮어쓰기, L 이후 추가 덮어쓰기 영역을 함께 관리할 수 있어야 한다.

예시:

```text
B~H 조건:
  매칭키=한국전력공사_3억미만
  입찰방식=적격심사
  종목=전기
  공고확인=배전공가

I~K 덮어쓰기:
  원발주처=한국전력공사(배전공가)
  적격발주처=한국전력공사
  적격평가기준_세부=한국전력공사(배전공가)_3억미만

L 이후 추가 덮어쓰기:
  지역제한=1
  추정가격기준=1
```

## 확인/실적 키워드 관리 구조

`공고확인`, `특수실적`, `특수실적_공통`은 웹 설정에서 관리하되 서로 합치지 않는다.
각각 다른 탭, 다른 저장소, 다른 파서맨 참조 대상으로 둔다.

권장 그룹:

```text
룰관리
  - 확인/실적 키워드
    - 공고확인
    - 특수실적
    - 특수실적_공통
```

첨부 파일:

| 탭 | 기준 파일 | 핵심 의미 |
|---|---|---|
| 공고확인 | `공고확인_2026-05-24.csv` | 사용자에게 확인할 태그와 2차 매핑 조건 |
| 특수실적 | `특수실적_2026-05-24.csv` | 종목/발주처별 실적 조건 |
| 특수실적_공통 | `특수실적_공통_2026-05-24.csv` | 기업인증, 신용평가, 장비, 협력업체 등 공통 조건 |

예시:

```text
공고확인 결과값=전자시담
  -> 공고확인 컬럼 후보

특수실적 결과값=철도배전선로[154Kv]_준공금액
  -> 특수실적 컬럼 후보

특수실적_공통 결과값=사회적기업
  -> 특수실적_공통 후보
```

주의:

```text
세 파일은 일부 컬럼명이 같아도 같은 테이블로 통합하지 않는다.
파서맨도 세 설정을 각각 따로 읽어 각기 다른 컬럼 후보로 기록한다.
종목, 발주처, 대분류는 결과값을 찾기 위한 검색·필터용 메타로 보고,
공고 row와 자동 매칭하는 조건으로 해석하지 않는다.
검색키워드가 빈값인 행은 사람이 직접 입력하는 값이다.
```

상세 기준:

```text
docs/20_특수실적_공고확인_설정데이터_초안.md를 따른다.
```

## 파서맨 관리 구조

파서맨은 파싱용공고문에서 값을 뽑는 결정형 룰 엔진이다.
기존 문서곡괭이 참고자료의 흐름은 참고하되, 새 시스템에서는 역할별로 다시 나눈다.

권장 그룹:

```text
룰관리
  - 파서맨
    - 룰 목록
    - 조건판단형태 가이드
    - 샘플 공고 테스트
    - 실패/충돌 리포트
```

예시:

```text
파서맨 룰:
대상컬럼=공고확인
조건판단형태=3_1
검색키워드=전자시담
고정값=전자시담

공고문:
본 공고는 전자시담 방식으로 진행합니다.

결과:
공고확인=전자시담
```

상세 기준:

```text
docs/21_파서맨_새구조_설계초안.md를 따른다.
```

## 남은 확인사항

아래 내용은 사용자 확인 후 확정한다.

```text
1. 여러 키워드가 동시에 매칭될 때 우선순위
```

질문 예시:

```text
원본 발주처=`한국전력공사 남부발전 관련 공고`처럼 여러 키워드가 동시에 걸리면
더 긴 키워드를 우선할까요, 아니면 `id`가 작은 행을 우선할까요?
```
