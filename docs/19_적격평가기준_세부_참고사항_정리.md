# 적격평가기준_세부 참고사항 정리

작성일: 2026-05-24

## 목적

이 문서는 기존 인계 문서에서 참고할 정보와 그대로 쓰면 안 되는 구현 세부사항을 분리한다.

참고자료는 현재 잘 작동하는 운영 흐름을 설명하지만,
우리 시스템은 새 구조로 다시 설계해야 한다.
따라서 파일명, 함수명, 화면 경로를 그대로 고정하지 않고
흐름, 실패 처리, 디버깅 방식, 주의사항만 구조 설계에 반영한다.

## 참고에서 가져올 핵심

### 1. 두 단계 매칭은 유효하다

적격평가기준_세부를 한 번에 찾지 않고,
기본 기준을 찾은 뒤 세부 조건으로 다시 덮는 구조는 유지 가치가 있다.

예시:

```text
1차 매칭:
한국전력공사 + 일반건설 + 추정가격 2.5억
  -> 한국전력공사_3억미만

2차 매칭:
공고확인=배전공가
  -> 한국전력공사(배전공가)_3억미만
```

### 2. 선행 컬럼이 중요하다

1차 매칭 전에 `일반_기타_전문`이 먼저 결정되어야 한다.
이는 종목 분류 결과가 적격심사평가기준 선택에 직접 들어가기 때문이다.

예시:

```text
종목=전기
일반_기타_전문=기타건설
  -> 1차 기준표에서 기타건설 금액 컬럼을 사용
```

### 3. 멱등성이 필요하다

같은 공고 row와 같은 룰로 여러 번 실행해도 결과가 같아야 한다.
사람 입력 후 2차분류를 다시 눌러도 이전 결과가 꼬이면 안 된다.

예시:

```text
공고확인=배전공가
1회 실행 -> 한국전력공사(배전공가)_3억미만
2회 실행 -> 한국전력공사(배전공가)_3억미만
```

### 4. 정상 SKIP과 미매칭은 구분해야 한다

적격심사 대상이 아닌 입찰은 빈값이 정상이다.
반대로 적격심사 기준을 찾아야 하는데 못 찾은 경우는 확인 대상이다.

예시:

```text
입찰방식=수의계약
  -> 적격평가기준_세부=빈문자열
  -> 정상 SKIP
```

예시:

```text
입찰방식=적격심사
추정가격=빈값
  -> 적격평가기준_세부=null
  -> 미매칭, 확인 필요
```

### 5. 실패 사유를 남겨야 한다

사람이 고칠 수 있으려면 단순히 빨간 셀만 보여주면 안 된다.
어느 단계에서 왜 실패했는지 남겨야 한다.

예시:

```text
1차 상태=missing
1차 사유=추정가격 없음 또는 0
처치=공고관리에서 추정가격 입력
```

예시:

```text
1차 상태=missing
1차 사유=원발주처 미등록: 경기도청
처치=1차 기준표에 경기도청 기준 행 추가
```

### 6. 미등록 발주처는 모아서 알려주는 편이 좋다

배치 중 하나씩 멈추면 수백, 수천 건 처리에서 흐름이 끊긴다.
배치 종료 후 미등록 발주처를 한 번에 모아 보여주는 방식이 맞다.

예시:

```text
미등록 발주처 2개 발견:
- 경기도청
- 인천시청
```

### 7. 원본 컬럼은 읽기 중심으로 다룬다

원발주처, 추정가격, 입력일 같은 원본 또는 사람 입력 컬럼은 함부로 변조하지 않는다.
분류 결과는 결과 컬럼과 메타 컬럼에 쓴다.

예시:

```text
추정가격=250,000,000
  -> 읽기만 함
  -> 적격평가기준_세부=한국전력공사_3억미만
  -> 매칭사유=1차 기준표 id=5
```

## 그대로 고정하지 않을 것

아래 항목은 참고자료의 현재 구현명일 뿐이다.
우리 시스템의 새 구조에서 그대로 고정하지 않는다.

| 참고 항목 | 그대로 쓰지 않는 이유 | 우리 구조에서의 해석 |
|---|---|---|
| `master_evaluationCriteria.json` | 직접 편집 구조는 웹 룰관리 목표와 충돌 | 1차 기준표 마스터 |
| `rules_orgCriteria.json` / `orgCriteria` | 파일명과 런타임 주입 방식은 구현 세부 | 2차 세부조건 마스터 |
| `rules_적격_skipKeywords.json` | 파일명 고정 불필요 | 기준대상/제외 입찰방식 마스터 |
| `applyPrimaryMatch` | 함수명은 기존 코드명 | 1차 기준 매칭 단계 |
| `applySecondaryMatch` | 함수명은 기존 코드명 | 2차 세부조건 매칭 단계 |
| `finalizeHook.ts` | 파일명은 구현 세부 | 분류 파이프라인 실행기 |
| `.cell-unmatched` | CSS명은 구현 세부 | 미매칭 표시 상태 |
| `Sonnet/Opus` 역할 분담 | 개발 운영 방식이지 제품 구조 아님 | 문서/개발 지침에 넣지 않음 |

## 새 구조로 재해석

참고사항을 우리 시스템에 맞게 재해석하면 아래 구조가 된다.

```text
적격심사기준 분류 엔진
  1. 대상판정
  2. 선행값 검증
  3. 발주처 정규화
  4. 1차 기준 매칭
  5. 2차 세부조건 매칭
  6. 결과 검증
  7. A4 호출 준비
  8. 디버그/처치정보 기록
```

## 권장 출력 메타

기존 `_v1_...` 컬럼명은 참고만 한다.
새 구조에서는 이름을 더 명확하게 둘 수 있다.

| 메타 | 의미 | 예시 |
|---|---|---|
| 적격_대상상태 | 대상, 제외, 확인필요 | `대상` |
| 적격_1차상태 | 성공, 미매칭, 제외 | `성공` |
| 적격_1차사유 | 1차 판단 이유 | `OK: 한국전력공사_3억미만` |
| 적격_2차상태 | 성공, 미매칭, 해당없음 | `성공` |
| 적격_2차사유 | 2차 판단 이유 | `공고확인=배전공가, rule id=1` |
| 적격_처치방법 | 사람이 고칠 방법 | `추정가격 입력 필요` |

예시:

```text
적격_1차상태=미매칭
적격_1차사유=원발주처 미등록: 경기도청
적격_처치방법=1차 기준표에 경기도청 기준 행 추가
```

## 빨간 셀 처치 흐름

미매칭은 사람이 고칠 수 있게 유형별 처치법을 연결한다.

| 미매칭 사유 | 처치 | 예시 |
|---|---|---|
| 원발주처 미등록 | 1차 기준표에 발주처 기준 행 추가 | `경기도청` 기준 추가 |
| 적용 단계 없음 | 1차 기준표에 해당 금액 단계 추가 | `30억미만` 행 추가 |
| 건설유형 없음 | 종목 분류 또는 사람 입력 보완 | `일반_기타_전문=기타건설` 입력 |
| 추정가격 없음 또는 0 | 추정가격 입력 | `추정가격=250000000` |
| 입찰방식 없음 | 입찰방식 입력 | `입찰방식=적격심사` |

## 구조 설계에 반영할 주의사항

1. SKIP 체크와 미매칭을 같은 빈값으로 처리하지 않는다.

예시:

```text
수의계약 -> 정상 제외
적격심사인데 추정가격 없음 -> 미매칭
```

2. 2차 매칭은 하드코딩하지 않고 런타임 룰 데이터를 주입한다.

예시:

```text
2차 세부조건 마스터를 수정하면 코드 수정 없이 다음 분류부터 반영
```

3. 조건이 비어 있는 2차 행은 전체 통과 후보로 볼 수 있지만 충돌 관리가 필요하다.

예시:

```text
매칭키=조달청_별표-1-5
입찰방식=간이종심제
조건이 같은데 결과가 2개이면 우선순위 또는 추가 조건 필요
```

4. 원본값과 결과값을 분리한다.

예시:

```text
수집 원발주처 후보=한국전력공사
2차 결과 원발주처=한국전력공사(배전공가)
두 값을 추적 가능하게 보관
```
