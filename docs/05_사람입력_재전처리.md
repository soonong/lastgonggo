# 사람입력과 재전처리

## 기본 원칙

사람이 입력한 값은 최우선으로 보호한다. 코드, 공고문 파싱, AI 후보값이 사람 입력값을 덮어쓰면 안 된다.

## 부분 재전처리

사람이 값을 수정하면 전체 데이터를 다시 처리하지 않고, 영향을 받는 컬럼만 다시 처리한다.

예시:

| 사람이 수정한 컬럼 | 다시 처리할 컬럼 | 다시 처리하지 않을 컬럼 |
|---|---|---|
| 공고명 | 공고분류, 종목그룹, 키워드 | 공고번호, 공고일자 |
| 발주처명 | 발주처표준명, 기관구분, 지역 | 원본 발주처명 |
| 기초금액 | 금액구간, 예산분류 | 공고명, 공고번호 |
| 추정가격 | 1차 적격심사평가기준, 적격평가기준_세부, A4 기준 row | 공고명, 공고번호 |
| 공고확인 | 2차 적격심사기준 매칭, 원발주처, 적격평가기준_세부 | 원본 공고문 |
| 원발주처 | 적격평가기준_세부, A4 기준 row, 종목만점실적, 종목세부JSON | 공고번호, 원본 수집값 |
| 적격평가기준_세부 | A4 기준 row, 종목만점실적, 종목세부JSON | 적격발주처, 공고번호 |
| 단독평가종목 | 종목세부JSON, 일반_기타_전문, 동일실적_평가여부 | 원본 종목 |

## 재전처리 절차

```text
1. 사람이 수정한 컬럼 확인
2. 재전처리 의존관계 룰 확인
3. 영향받는 컬럼만 재계산
4. 사람 입력값은 보호
5. 변경된 값과 이유 기록
```

## 적격심사기준 재전처리

적격심사기준 관련 컬럼은 순서가 꼬이면 뒤 계산이 모두 틀어진다.
따라서 사람이 `원발주처`, `적격평가기준_세부`, `단독평가종목` 같은 값을 수정하면 필요한 컬럼만 다시 계산한다.

예시:

```text
사람 입력: `원발주처=한국전력공사(배전)`
재전처리:
1. 1차 적격심사평가기준 다시 찾기
2. 2차 적격심사기준 매칭 다시 실행
3. A4 적격심사세부기준 다시 호출
4. 실적배수와 인정기간 다시 추출
5. 종목만점실적 다시 계산
6. 종목세부JSON 다시 생성
```

예시:

```text
사람 입력: `공고확인=배전공가`
재전처리:
1. 2차 적격심사기준 매칭 다시 실행
2. 원발주처=한국전력공사(배전공가)로 덮기
3. 적격평가기준_세부=한국전력공사(배전공가)_3억미만으로 덮기
4. A4 이후 파생값 다시 계산
```

## 예시

상황:

```text
API 업종: 기타
공고문 후보: 전기공사업
사람 입력: 전기공사업
```

처리:

```text
최종 업종은 `전기공사업`으로 확정한다.
업종에 의존하는 `종목그룹`, `공고분류`만 다시 처리한다.
API 원본값 `기타`는 보존한다.
```
