# 전처리 파이프라인 전수조사 (2026-05-30)

## 조사 목적

`입찰공고 원본`, `입찰공고 전처리`, `공고관리`, `2차분류 완료`가 서로 다른 일을 해야 하는데, 현재 일부 처리가 `preprocessRows` 한 함수에 모여 있어 단계 책임이 섞일 수 있다. 이 문서는 실제 버튼 호출 기준으로 무엇이 실행되는지, 충돌 후보가 무엇인지, 처리 상태가 무엇인지 기록한다.

## 버튼별 실제 호출

| 화면 | 버튼 | 실제 함수 | 내부 호출 | 현재 역할 |
|---|---|---|---|---|
| 입찰공고 원본 | API 호출 | `loadA1` | `/api/bid` | A1 서버공고 수집, 자동수집여부/입력일/종목 필터, 표시형식 검사 |
| 입찰공고 원본 | 곡괭이전나라시 | `runBeforeNarashi` | `preprocessRows(..., skipSpecialConditionRouter=true, skipSiteRegionNormalization=true)` | 원본 row를 50건씩 전처리하여 입찰공고 전처리로 이동 |
| 입찰공고 전처리 | 문서곡괭이질 시작 | `runPreprocess` | `fetchA3Parser` -> `mergeParserFields` -> `preprocessRows` | A3 문서곡괭이 결과를 row에 병합 후 후속 전처리 |
| 공고관리 | 재전처리 | `saveHumanEdit` | `reprocessHumanRow` -> `preprocessRows` | 사람이 고친 row 재전처리 |
| 공고관리 | 2차분류 | `finalizeRows` | `preprocessRows` | 선택 row를 재전처리 후 2차분류 완료로 이동 |

## `preprocessRows` 내부 처리

현재 `preprocessRows`는 아래 처리를 순서대로 실행한다.

1. 삭제키워드 제외
2. raw 문자열/금액 기본 정리
3. 공사명 괄호/대괄호 정리
4. 특수조건 라우팅
5. 지역제한 정규화
6. 공사현장 정규화
7. 기본 계산
8. 표준 컬럼 빈값 추가
9. 적격심사 1차/2차 디버그/매칭
10. 검증 상태 기록

## 곡괭이전나라시에서 제외 처리 완료

| 항목 | 상태 | 이유 |
|---|---|---|
| 특수조건 라우팅 | 제외 완료 | 문서곡괭이 이후 또는 사람 입력 저장/2차분류 시점의 규칙 |
| 공사현장 정규화 | 제외 완료 | 공사현장은 사람이 입력한 뒤 정규화해야 함 |
| local-sample API | 제거 완료 | 샘플 버튼 제거 후 호출처 없음 |

## 남은 충돌 후보

아래 항목은 아직 곡괭이전나라시에서 실행된다. 단계 정의상 정말 원본 단계에서 해야 하는지 확인이 필요하다.

| 후보 | 현재 실행 위치 | 충돌 이유 | 권장 처리 |
|---|---|---|---|
| 종목묶기/단독평가종목/종목세부JSON | 곡괭이전나라시 포함 | 문서곡괭이로 종목/주력분야를 보완한 뒤 다시 해야 정확함 | 곡괭이후나라시로 이동 후보 |
| 투찰금액 계산 | 곡괭이전나라시 포함 | A값/투찰율/기초금액이 문서곡괭이로 보완될 수 있음 | 문서곡괭이 이후 계산 후보 |
| 적격심사 1차/2차 매칭 | 곡괭이전나라시 포함 | 원발주처, 공고확인, 특수실적, 종목 정보가 아직 확정 전 | 곡괭이후나라시 또는 2차분류 직전으로 이동 후보 |
| 검증 상태 | 곡괭이전나라시 포함 | 문서곡괭이 전 빈값을 오류처럼 볼 수 있음 | 단계별 검증으로 분리 후보 |

## 다음 정리 방향

`preprocessRows`를 하나의 만능 함수로 유지하면 계속 단계 충돌이 난다. 다음 구조로 분리하는 것이 안전하다.

```text
runBeforeNarashi
  -> normalizeRaw
  -> applyDeleteKeywords
  -> applyBracketRules
  -> applyColumnDefaults

runAfterDocumentPickaxe
  -> mergeParserFields
  -> routeSpecialCondition
  -> normalizeRegionLimit
  -> buildJongmok
  -> runCalculations
  -> matchEvaluationCriteria
  -> validateAfterPickaxe

runHumanReprocess
  -> normalizeHumanInputFields
  -> routeSpecialCondition
  -> normalizeSiteRegion
  -> recomputeDependentFieldsOnly
  -> matchEvaluationCriteria
```

## 확인 필요 질문

1. 곡괭이전나라시에서 `지역제한` 정규화는 유지할까?
   예: 서버공고 지역제한=`구리`이면 원본 단계에서 `경기,구리시`로 바꿀지, 문서곡괭이 이후로 미룰지.
2. 곡괭이전나라시에서 `종목조건없음` 처리만은 유지할까?
   예: 서버공고 종목=`업종조건없음`이면 원본 단계에서 빈값으로 둘지.
3. 곡괭이전나라시에서 표준 컬럼 빈값 추가는 유지할까?
   예: 전체 컬럼 표를 바로 보여주기 위해 빈 컬럼을 생성하는 용도.
