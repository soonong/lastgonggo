# 전처리 파이프라인 전수조사 (2026-05-30)

## 조사 목적

`입찰공고 원본`, `입찰공고 전처리`, `공고관리`, `2차분류 완료`가 서로 다른 일을 해야 하는데, 현재 일부 처리가 `preprocessRows` 한 함수에 모여 있어 단계 책임이 섞일 수 있다. 이 문서는 실제 버튼 호출 기준으로 무엇이 실행되는지, 충돌 후보가 무엇인지, 처리 상태가 무엇인지 기록한다.

## 버튼별 실제 호출

| 화면 | 버튼 | 실제 함수 | 내부 호출 | 현재 역할 |
|---|---|---|---|---|
| 입찰공고 원본 | API 호출 | `loadA1` | `/api/bid` | A1 서버공고 수집, 자동수집여부/입력일/종목 필터, 표시형식 검사 |
| 입찰공고 원본 | 곡괭이전나라시 | `runBeforeNarashi` | `preprocessBeforeNarashiRows` | 원본 row를 50건씩 원본 정리만 해서 입찰공고 전처리로 이동 |
| 입찰공고 전처리 | 문서곡괭이질 시작 | `runPreprocess` | `fetchA3Parser` -> `mergeParserFields` -> `preprocessAfterDocumentRows` | A3 문서곡괭이 결과를 row에 병합 후 1차/후나라시/2차 흐름 실행 |
| 공고관리 | 재전처리 | `saveHumanEdit` | `reprocessHumanRow` -> `reprocessHumanRows` | 사람이 고친 row만 재전처리 |
| 공고관리 | 2차분류 | `finalizeRows` | `reprocessHumanRows` | 선택 row만 2차 매칭 계열을 재실행 후 2차분류 완료로 이동 |

## `preprocessRows` 내부 처리

`preprocessRows` 하나에 몰아넣던 구조를 단계별 함수로 분리했다.

| 함수 | 실행 범위 |
|---|---|
| `preprocessBeforeNarashiRows` | 삭제키워드, raw 정리, 공사명 괄호/대괄호 정리, 표준 컬럼 빈값 추가 |
| `preprocessAfterDocumentRows` | 문서곡괭이 이후 특수조건 라우팅, 지역제한 정규화, 적격 1차, 종목묶기/투찰계산 등 후나라시, 적격 2차, 검증 |
| `reprocessHumanRows` | 사람 입력 row의 특수조건 라우팅, 지역/공사현장 정규화, 의존 계산, 적격 2차 재실행, 검증 |

## 곡괭이전나라시에서 제외 처리 완료

| 항목 | 상태 | 이유 |
|---|---|---|
| 특수조건 라우팅 | 제외 완료 | 문서곡괭이 이후 또는 사람 입력 저장/2차분류 시점의 규칙 |
| 공사현장 정규화 | 제외 완료 | 공사현장은 사람이 입력한 뒤 정규화해야 함 |
| local-sample API | 제거 완료 | 샘플 버튼 제거 후 호출처 없음 |

## 남은 충돌 후보

아래 항목은 아직 곡괭이전나라시에서 실행된다. 단계 정의상 정말 원본 단계에서 해야 하는지 확인이 필요하다.

| 후보 | 현재 실행 위치 | 충돌 이유 | 권장 처리 |
|---|---|---|---|
| 종목묶기/단독평가종목/종목세부JSON | 곡괭이전나라시에서 제거 완료 | 문서곡괭이 이후 후나라시에서 실행 |
| 투찰금액 계산 | 곡괭이전나라시에서 제거 완료 | 문서곡괭이 이후 후나라시에서 실행 |
| 적격심사 1차/2차 매칭 | 곡괭이전나라시에서 제거 완료 | 문서곡괭이 후 1차, 후나라시 후 2차 실행 |
| 검증 상태 | 곡괭이전나라시에서 제거 완료 | 문서곡괭이 후/공고관리 단계에서 실행 |

## 다음 정리 방향

현재 코드 구조는 아래처럼 분리한다.

```text
runBeforeNarashi / preprocessBeforeNarashiRows
  -> 삭제키워드
  -> raw 정리
  -> 공사명 괄호/대괄호 정리
  -> 표준 컬럼 빈값 추가

runPreprocess / preprocessAfterDocumentRows
  -> 문서곡괭이 결과 병합
  -> 특수조건 라우팅
  -> 지역제한 정규화
  -> 적격 1차
  -> 종목묶기/투찰계산
  -> 적격 2차
  -> 검증

saveHumanEdit/finalizeRows / reprocessHumanRows
  -> 사람 입력값 정리
  -> 특수조건 라우팅
  -> 지역제한/공사현장 정규화
  -> 의존 계산
  -> 적격 2차 재실행
  -> 검증
```

## 현재 결정

- 곡괭이전나라시에서는 `지역제한`과 `공사현장` 정규화를 하지 않는다.
- 곡괭이전나라시에서는 `종목묶기`, `투찰금액 계산`, `적격심사 매칭`, `검증 상태 기록`을 하지 않는다.
- 곡괭이전나라시에서는 표준 컬럼을 전체 표에 보여주기 위한 빈값 추가만 유지한다.
- `종목조건없음` 처리는 raw 값 정리의 일부로 유지한다. 서버공고의 `업종조건없음` 같은 값은 표준 종목값이 아니므로 빈값으로 만든다.
