### 결정: 서버공고 컬럼 표시 기준

서버공고에서 내려받은 모든 컬럼을 엑셀처럼 가로로 표시한다.

이유:

```text
수집 원본의 일부 컬럼만 보여주면 빈값 보완, 사람 입력, 최종분류 단계에서 어떤 값이 누락되었는지 확인하기 어렵다.
서버공고의 항목명과 값 형식을 먼저 고정한 뒤 사용자가 만든 컬럼 정의와 비교해야 한다.
```

예시:

```text
서버공고 응답에 `공고번호`, `기초금액`, `종목세부JSON`, `종목금액합계`가 있으면
각 항목을 숨기지 않고 한 행의 가로 컬럼으로 표시한다.
```

검증 결과:

```text
2026-05-22 기준 서버공고 컬럼은 118개다.
서버공고 건수는 실시간 수집 목록이라 실행 시점에 따라 3348건에서 3356건처럼 변동될 수 있다.
`api 컬럼형식.xlsx`는 47행 중 41개만 서버공고 컬럼과 일치한다.
`apiColumnFormat.json`은 129행 중 117개가 서버공고 컬럼과 일치하고, 서버공고 컬럼 중 `종목금액합계`만 빠져 있다.
`API_컬럼형식_2026-05-22.csv`도 129행 중 117개가 서버공고 컬럼과 일치하고, 서버공고 컬럼 중 `종목금액합계`만 빠져 있다.
```

확정 사항:

```text
`종목금액합계`는 현재 쓰지 않으므로 표준 컬럼 CSV에 추가하지 않는다.
`입력일`은 날짜까지만 표시한다.
`난이도계수`는 소수점 둘째자리까지 표시한다.
`상호진출여부`는 0/1 숫자로 유지한다.
CSV에만 있는 12개 컬럼은 후처리/검증용 컬럼으로 유지한다.
```

예시:

```text
`입력일=2026-04-20 15:17:00`은 화면과 엑셀에서 `2026-04-20`까지만 표시한다.
`난이도계수=0.85`는 `0.85`처럼 둘째자리까지 표시한다.
`상호진출여부=1`은 체크박스가 아니라 숫자 `1`로 둔다.
```

### 결정: 전처리 기준 파일 관리

전처리 기준이 되는 파일은 웹 설정 화면에서 추가, 변경, 수정할 수 있게 한다.

이유:

```text
적격발주처, 종목 매핑, 공고확인 선택목록, 특수실적 선택목록 같은 기준은 계속 바뀔 수 있다.
코드에 고정하면 운영 중 변경이 어렵고, 사용자 설명과 실제 룰이 어긋날 수 있다.
```

예시:

```text
`한국가스공사 부산·경남지역본부 -> 한국가스공사` 같은 적격발주처 매핑은
설정 화면에서 파일 또는 테이블로 추가/수정한다.
```

확인된 파일:

```text
`적격발주처_변경_2026-05-22.csv`는 `id`, `발주처 키워드`, `매핑 대상` 3컬럼 구조다.
예: `한국남부발전 -> 조달청`
`발주처 키워드`는 포함검색으로 매칭한다.
`매핑 대상`은 `적격발주처`를 덮지 않고 `원발주처`에 넣는다.
상세 기준은 `docs/rules/설정마스터룰.md`를 따른다.
```

### 결정: 적격심사기준 컬럼 흐름

공고분류 전처리는 해당 공고의 적격심사기준을 찾기 위한 작업이다.

핵심 컬럼은 아래처럼 구분한다.

| 컬럼 | 의미 |
|---|---|
| 적격발주처 | 서버공고에 컬럼으로 올 수 있지만 값이 항상 채워진다고 보지 않는다. 값이 있으면 홈페이지 연결 기준 후보로 사용 |
| 원발주처 | 실제 평가에 사용할 적격심사기준 발주처 |
| 적격평가기준_세부 | A4 적격심사세부기준 API를 호출하기 위한 세부 기준명 |

이유:

```text
발주처 홈페이지와 실제 평가 기준이 항상 같지는 않다.
예를 들어 한국전력공사 홈페이지 공고라도 실제 심사기준은 조달청일 수 있다.
또한 같은 발주처 내부에서도 배전, 가공송전처럼 특수 기준이 나뉠 수 있다.
```

예시:

```text
서버공고 또는 전처리 후보: `적격발주처=한국전력공사`
실제 평가 기준: `원발주처=한국전력공사(배전)`
A4 호출 기준: `적격평가기준_세부=한국전력공사(배전)_10억미만`
```

상세 규칙:

```text
docs/rules/적격심사기준룰.md를 따른다.
```

### 결정: 종목룰 분리

`종목`, `단독평가종목`, `종목세부JSON`은 한몸으로 보고 별도 종목룰 문서에서 관리한다.

이유:

```text
종목은 건설유형, 상위종목, 대업종/주력종목, 상호진출, 단독평가종목 분해, 종목세부JSON 전개가 연결된 복잡한 규칙이다.
컬럼룰 문서 안에 모두 넣으면 유지보수가 어렵다.
```

예시:

```text
`건축 + 토목 -> 토건`,
`금속구조물 + 지붕판금 -> 금속창호지붕`,
`단독평가종목=(토목,산업환경설비) -> 종목세부JSON 객체 2개`
같은 규칙은 `docs/rules/종목룰.md`에 누적한다.
```

### 결정: 종목 업무 용어 통일

업무 문서와 화면 설명에서는 `업종`을 `종목`으로 통일한다.
단, CSV 원본 컬럼명이 `업종`, `주력업종`, `상위업종`, `대체업종`인 경우에는 원본 컬럼명을 보존하고 업무 해석을 함께 적는다.

이유:

```text
입찰 참여 자격을 판단하는 기준은 앞으로 종목으로 부른다.
같은 개념을 업종과 종목으로 섞어 쓰면 종목, 단독평가종목, 종목세부JSON 규칙이 흔들릴 수 있다.
```

예시:

```text
종목_매핑 CSV의 `업종=금속창호지붕`, `주력업종=금속구조물`은
업무상 `종목=금속창호지붕`, `주력종목=금속구조물`로 읽는다.
```

### 결정: 종목묶기룰 분리

표시용 `종목`, 심사용 `단독평가종목`, `종목세부JSON`을 만드는 괄호/쉼표/슬래시 규칙은 `종목룰.md`에서 분리해 `종목묶기룰.md`로 관리한다.

이유:

```text
종목묶기는 상위종목, 전문건설 주력종목, 상호진출, 종목모두보유, 공동도급 조건이 함께 작동하는 복잡한 규칙이다.
기본 개념 문서와 분리해야 나중에 로직 변경과 예시 검증이 쉽다.
```

예시:

```text
`건축/토목 -> (건축,토목)/토건`
`토공/토목, 상호진출=1 -> 종목=지반조성포장 / 토목 / 토건`
같은 상세 표기 규칙은 `docs/rules/종목묶기룰.md`에 기록한다.
```

### 결정: 종목묶기 케이스 우선 작업 방식

종목묶기는 사용자가 제공한 케이스를 바탕으로 전체 케이스 매트릭스를 먼저 만들고, 명확해질 때까지 계속 보강한다.
질문은 바로 정답을 요구하는 방식이 아니라, 비어 있는 케이스를 예시와 함께 확인하는 방식으로 한다.

이유:

```text
종목묶기는 조합 수가 많고, 일부 예시만 보고 구현하면 상호진출, 상위종목, 공동도급, 대체종목에서 누락이 생길 수 있다.
따라서 구현 전 케이스 ID, 입력 조건, 기대 종목, 기대 단독평가종목, 상태를 먼저 고정해야 한다.
```

예시:

```text
JM-N0-G-005: 토목/토공/포장
  -> 종목=(토목,지반조성포장)/(토건,지반조성포장)
  -> 단독평가종목=(토목,지반조성포장_토공,지반조성포장_포장)

JM-GAP-001: 종목모두보유=1인 실제 공고
  -> 아직 비어 있는 케이스로 두고 추가 예시를 받아 확정한다.
```

상세 기준:

```text
docs/rules/종목묶기_케이스매트릭스.md를 따른다.
```

## 2026-05-24

### 결정: 종목모두보유와 공동도급 포지션

`종목모두보유`, `공동이행`, `분담이행`, `주계약자관리`는 종목묶기에서 같은 포지션으로 처리한다.

이유:

```text
이 조건들은 모두 여러 종목을 하나의 참여 조건 묶음으로 해석해야 하는 축이다.
따라서 하위 종목을 괄호로 묶고, 상위종목은 별도 옵션으로 만든다.
```

예시:

```text
종목모두보유=1, 입력=건축/토목
  -> 종목=(건축,토목)/토건

종목모두보유=0, 입력=건축/토목
  -> 종목=건축/토목/토건

공동도급형태=분담이행, 입력=건축/전기
  -> 종목=(건축,전기)/(토건,전기)
```

### 결정: 상호진출 단독평가종목의 토건 제외

`토건`은 표시용 종목에는 나올 수 있지만, 어디서든 단독평가종목에는 넣지 않는다.

이유:

```text
토건은 단독평가종목에서 평가하지 않으므로 평가종목으로 만들면 안 된다.
```

예시:

```text
입력=토공/토목, 상호진출=1
  -> 종목=지반조성포장 / 토목 / 토건
  -> 단독평가종목=지반조성포장_토공 / 토목_지반조성포장_토공
  -> 토건_지반조성포장_토공은 만들지 않는다.
```

### 결정: 상호진출 접미사는 매핑표 기준

상호진출에서 전문종목과 일반종목을 연결하는 접미사는 별도 매핑표를 근거로 만든다.
여러 일반종목 또는 전문종목이 모두 매핑되면 둘 중 하나를 사용할 수 있지만, 묶음 전체를 접미사로 붙이는 케이스는 없다.

이유:

```text
상호진출 접미사는 단순히 첫 종목을 붙이는 규칙이 아니라, 전문종목과 일반종목의 대응 관계를 따라야 한다.
```

예시:

```text
입력=건축/토목/실내건축, 상호진출=1
  -> 단독평가종목=실내건축_건축
  -> 실내건축_(건축,토목)은 만들지 않는다.
```

### 결정: 대체종목 자동 인정

대체종목은 참여 가능 면허 매칭에서 자동 인정한다.
표시용 종목에는 원 종목과 대체종목을 슬래시로 함께 보여준다.
단독평가종목은 원 종목 기준을 유지한다.

이유:

```text
대체종목은 회사 면허와 공고 종목을 매칭 가능하게 하기 위한 보조 인정 관계다.
화면에는 대체 인정 관계가 보여야 하지만, 심사 기준까지 대체종목으로 바꾸면 원 공고 기준이 흐려질 수 있다.
```

예시:

```text
기계가스설비 -> 가스난방
회사 면허=가스난방
  -> 기계가스설비 공고 매칭 가능
  -> 표시용 종목=기계가스설비/가스난방
  -> 단독평가종목=기계가스설비 또는 기계가스설비_기계설비
```

### 결정: 미분류와 종목조건없음 처리

`미분류`는 시설 종목 매칭에서 제외한다.
`종목조건없음`은 종목과 단독평가종목 모두 빈값을 반환한다.
단, 원본 CSV나 서버 데이터에 `업종조건없음`이라는 표현이 들어오면 같은 의미로 읽는다.

이유:

```text
미분류는 일반/전문/기타건설의 시설 종목으로 확정할 수 없는 값이다.
종목조건없음은 실제 종목명이 아니라 종목 조건이 없다는 상태다.
```

예시:

```text
해외건설업(전문건설업-도장공사업)
  -> 일반_기타_전문=미분류
  -> 시설 종목 매칭 제외

종목조건없음 또는 원본 업종조건없음
  -> 종목=
  -> 단독평가종목=
```

### 결정: 종목 매핑 중복은 전처리 정규화로 해소

원본문자만 보고 중복 매핑을 선택하지 않는다.
수집/파싱 단계에서 `종목`, `전문건설_주력분야`, `일반_기타_전문`을 먼저 만든 뒤, 그 정규화된 값을 기준으로 종목묶기를 한다.

이유:

```text
`포장`, `토공`, `소방` 같은 원본문자는 매핑표에서 여러 결과에 걸릴 수 있지만,
실제 공고는 전처리 후 대업종과 주력분야로 분리되므로 종목묶기 단계에서는 충돌이 아니다.
```

예시:

```text
공고문=포장공사업
  -> 종목=지반조성포장
  -> 전문건설_주력분야=포장
  -> 단독평가종목=지반조성포장_포장

공고문=포장공사업/토공사업, 종목모두보유=0
  -> 종목=지반조성포장
  -> 단독평가종목=지반조성포장_포장/지반조성포장_토공

공고문=포장공사업/토공사업, 종목모두보유=1
  -> 종목=지반조성포장
  -> 단독평가종목=(지반조성포장_포장,지반조성포장_토공)

공고문=소방
  -> 전처리에서 전문소방으로 변환
```

### 결정: 대체종목 표시와 매칭

대체종목은 참여 가능 면허 매칭에서 자동 인정하고, 표시용 종목에는 원 종목과 대체종목을 슬래시로 함께 표시한다.
단독평가종목은 원 종목 기준을 유지한다.

이유:

```text
사용자가 화면에서 대체 인정 관계를 바로 볼 수 있어야 한다.
하지만 심사용 단독평가종목까지 대체종목으로 바꾸면 원 공고의 평가 기준이 흐려질 수 있다.
```

예시:

```text
기계가스설비 -> 가스난방
  -> 회사 면허=가스난방이면 매칭 가능
  -> 표시용 종목=기계가스설비/가스난방
  -> 단독평가종목=기계가스설비 또는 기계가스설비_기계설비
```

### 결정: 상호진출 상대종목 관리 위치

상호진출 상대종목은 종목매핑 마스터에서 함께 관리한다.
1차 컬럼명은 `상호진출_상대종목`으로 둔다.

이유:

```text
상호진출 매핑은 종목의 속성에 가깝다.
별도 파일로 분리하면 종목 매핑과 상호진출 기준이 따로 변경되어 충돌할 수 있다.
```

예시:

```text
건축 -> 실내건축, 금속창호지붕, 도장습식방수석공, 비계구조, 철콘, 지반조성포장
토목 -> 지반조성포장, 상하수도, 철도궤도, 수중준설, 승강기삭도, 철콘, 비계구조
산업환경설비 -> 기계가스설비, 상하수도, 수중준설, 가스난방
조경 -> 조경식재시설물, 지반조성포장
```

주의:

```text
철콘처럼 건축과 토목 양쪽에 모두 매핑되는 종목은 입력순서 기준 첫 매칭을 자동값으로 둔다.
이 값은 사람이 수정할 수 있어야 한다.
```

### 결정: 종목세부JSON 전개 방식

종목세부JSON은 `단독평가종목`의 구분자 의미를 보존하지 않는다.
슬래시, 괄호, 쉼표를 해석한 뒤 앞에서부터 순서대로 모든 종목을 넣는다.

이유:

```text
종목세부JSON은 평가 계산에 쓸 종목 단위 목록이다.
표시 문자열의 옵션/필수 묶음 기호를 JSON에 그대로 들고 갈 필요가 없다.
```

예시:

```text
단독평가종목=a/(b,c)/d
  -> 종목세부JSON 순서=a, b, c, d

단독평가종목=(지반조성포장_포장,지반조성포장_토공)
  -> 종목세부JSON 순서=지반조성포장_포장, 지반조성포장_토공
```

### 결정: 종목 입력순서 보존

종목 표기와 단독평가종목 표기는 입력순서를 보존한다.

이유:

```text
입력의 맨 앞 종목은 주공종 판단에 쓰이고, 사용자가 공고문에서 본 순서와 화면 표시 순서가 어긋나면 검증이 어려워진다.
```

예시:

```text
입력=전기/건축
  -> 종목=(전기,건축)/(전기,토건)

입력=건축/전기
  -> 종목=(건축,전기)/(토건,전기)
```

### 결정: 종목세부JSON 금액 기준

종목세부JSON의 금액 기준은 `평가기준금액`으로 한다.
평가기준금액은 적격심사세부기준의 배수적용기준에 따라 추정가격, 기초금액, 종목별 금액 등이 될 수 있다.

이유:

```text
종목만점실적과 세부 평가금액은 단순히 추정가격만 보고 계산하면 틀릴 수 있다.
적격심사세부기준이 어떤 금액을 기준으로 삼는지 먼저 확인해야 한다.
```

예시:

```text
단독평가종목=지반조성포장_포장/지반조성포장_토공
평가기준금액=1억
  -> 지반조성포장_포장 평가기준금액=1억
  -> 지반조성포장_토공 평가기준금액=1억
```

### 결정: 괄호 묶음 종목별 금액

괄호 묶음은 각 종목을 모두 보유해야 하고, 평가도 각 종목별로 필요하다.
종목별 금액이 있으면 그 금액을 그대로 쓴다.
금액이 없으면 시스템 기본값은 첫 번째 종목에 전체 평가기준금액을 넣고, 사람이 수정할 수 있게 둔다.

이유:

```text
비율 계산으로 임의 배분하면 정확하지 않을 수 있다.
공고문에 금액이 없고 비율만 있을 때만 평가기준금액에서 역산한다.
```

예시:

```text
단독평가종목=(토목,전기)
토목 금액=8억
전기 금액=2억
  -> 토목 평가기준금액=8억
  -> 전기 평가기준금액=2억

단독평가종목=(토목,전기)
평가기준금액=10억
종목별 금액 없음
  -> 첫 번째 종목 토목에 10억 입력
  -> 사람이 전기 금액을 수정할 수 있게 둠
```

### 결정: 종목만점실적 계산식

종목만점실적은 `평가기준금액 x 만점배수`로 계산한다.

이유:

```text
만점배수는 A4 적격심사세부기준에서 가져오고,
평가기준금액은 배수적용기준에 따라 달라지기 때문이다.
```

예시:

```text
평가기준금액=5억
만점배수=1.0
  -> 종목만점실적=5억
```

주의:

```text
현재는 주공종 기준으로 A4 적격심사세부기준을 불러온다.
종목별 평가기준금액이 달라 개별 호출이 필요해 보이는 케이스는 향후 확장 후보로 메모한다.
```

### 결정: 단독평가종목 중복 알림

단독평가종목에 완전히 같은 옵션이 반복되면 조용히 중복 제거하지 않고 알림을 띄운다.
일부 종목이 겹치더라도 옵션 전체가 다르면 정상으로 보고 순서대로 전개한다.

이유:

```text
단독평가종목=전기/전기 또는 (a,b)/(a,b) 같은 값은 정상 중복이 아니라 조립 로직이 꼬인 신호다.
그냥 dedupe하면 오류를 숨기게 된다.
반대로 (a,b)/(a,c)는 같은 a가 반복되어도 서로 다른 옵션이므로 유지해야 한다.
```

예시:

```text
단독평가종목=전기/전기
  -> 알림: 단독평가종목 중복 반복 확인 필요

단독평가종목=(a,b)/(a,c)
  -> 종목세부JSON 순서=a, b, a, c
```

### 결정: 사람 입력 후 종목 관련 재전처리

사람이 종목이나 전문건설_주력분야를 수정하면 종목 관련 파생 컬럼만 다시 처리한다.
상호진출 자동값이 틀리면 별도 선택 컬럼을 만들지 않고 `단독평가종목`을 직접 수정한다.

이유:

```text
사람이 입력한 원값은 보호해야 하지만,
단독평가종목, 종목세부JSON, 종목별 평가기준금액, 종목만점실적은 그 값에 의존하기 때문이다.
상호진출의 상대종목 선택은 사람이 최종 확인해야 하는 영역이므로 직접 수정이 가장 명확하다.
```

예시:

```text
사람이 전문건설_주력분야를 포장에서 토공으로 수정
  -> 전문건설_주력분야=토공은 보호
  -> 단독평가종목=지반조성포장_토공
  -> 종목세부JSON과 종목만점실적 재계산

상호진출 자동값=철콘_건축
사람이 단독평가종목=철콘_토목으로 직접 수정
  -> 단독평가종목은 보호
  -> 종목세부JSON, 종목별 평가기준금액, 종목만점실적 재계산
```

### 결정: 종목별 금액 빈값 처리

괄호 묶음에서 종목별 금액이 없으면 첫 번째 종목에 전체 평가기준금액을 넣고, 나머지 종목의 금액은 `0`이 아니라 빈값으로 둔다.
비율 역산값에 별도 산출근거 메모를 자동 생성하지 않는다.

이유:

```text
빈값은 사람이 확인해서 채워야 할 대상임을 보여준다.
0은 실제 금액이 0인 것처럼 오해될 수 있다.
```

예시:

```text
단독평가종목=(토목,전기)
평가기준금액=10억
종목별 금액 없음
  -> 토목 평가기준금액=10억
  -> 전기 평가기준금액=빈값
```

### 결정: 지역매핑 기준

지역제한, 공사현장, 검색용현장은 `지역디비.xlsx`를 기준으로 전처리한다.
지역디비는 웹 설정의 지역 설정 탭에서 추가, 변경, 수정할 수 있어야 한다.

이유:

```text
발주처마다 지역명을 `경기도`, `경기`, `구리`, `구리시`처럼 다르게 쓰므로
코드 고정이 아니라 설정 가능한 마스터로 관리해야 한다.
```

예시:

```text
수집값=경기도
  -> 광역시/도-전처리전에서 매칭
  -> 지역제한=경기

수집값=구리
  -> 시군구-홈페이지용 또는 시군구 매핑
  -> 지역제한=경기,구리시
```

### 결정: 지역제한 표시 형식

지역제한은 광역시/도만 있으면 광역 표준값만 표시하고, 시군구가 있으면 `광역,시군구`로 표시한다.
여러 지역은 `/`로 구분한다.

이유:

```text
지역제한은 최종 엑셀과 화면에서 사람이 바로 읽어야 하므로 짧고 일관된 형식이 필요하다.
```

예시:

```text
서울/경기도/강원
  -> 서울/경기/강원

남양주/구리
  -> 경기,남양주시/경기,구리시

분당구
  -> 경기,성남시

전국/울산
  -> 전국/울산
```

### 결정: 지역 매칭 보수 원칙

지역값은 `/`로 먼저 나눈 뒤 각 token을 매핑한다.
여러 광역에 같은 시군구명이 있으면 자동 확정하지 않는다.

이유:

```text
중구, 동구, 서구, 남구, 북구, 강서구, 고성군처럼 여러 광역에 존재하는 지역명이 있다.
광역 힌트 없이 자동 선택하면 잘못된 지역제한이 만들어질 수 있다.
```

예시:

```text
수집값=중구
  -> 서울 중구, 부산 중구, 대구 중구, 인천 중구, 대전 중구, 울산 중구 가능
  -> 자동 확정하지 않고 검토필요
```

### 결정: 공사현장 특수값

공사현장은 지역명뿐 아니라 사업소 형태 특수값도 같은 컬럼에 그대로 보관한다.
별도 컬럼은 늘리지 않는다.

이유:

```text
배전사업소, 송전사업소, 폐광지역진흥지구는 일반 지역명과 겹치지 않는다.
발전소지역은 파일이 들어오면 같은 특수지역 계열로 검토한다.
공사현장 계산과 표시를 위해 같은 컬럼에서 관리하는 편이 단순하다.
```

예시:

```text
공사현장=폐광지역진흥지구
  -> 공사현장=폐광지역진흥지구

공사현장=남서울본부/강남지사
  -> 공사현장=남서울본부/강남지사
```

### 결정: 하위구 지역 표기

하위구나 동읍이 매칭되더라도 최종 지역제한 표시는 상위 시군구까지만 쓴다.

이유:

```text
지역제한의 기본 표시는 광역 또는 광역,시군구 구조다.
하위구는 매칭을 돕는 특수값으로 쓰고, 최종 표시는 시군구 기준으로 맞춘다.
```

예시:

```text
수집값=분당구
  -> 하위구=분당구
  -> 상위 시군구=성남시
  -> 지역제한=경기,성남시
```

### 결정: 전국과 지역 혼합값

`전국`과 특정 지역이 함께 들어온 값은 `전국`으로 축약하지 않고 `/` 구분값을 유지한다.

이유:

```text
원문이 `전국/울산`처럼 들어온 경우에는 발주처가 둘을 구분해서 제공한 값일 수 있다.
임의로 `전국` 하나로 줄이면 원문 정보가 사라진다.
```

예시:

```text
수집값=전국/울산
  -> 지역제한=전국/울산
```

### 결정: 복합 지역 token 처리

한 token 안에 광역과 시군구가 같이 들어오면 각각 쪼개서 전처리한 뒤 다시 합친다.
공고문 자체가 지역을 잘못 적는 경우가 있으므로, 광역과 시군구의 상하관계가 맞는지 확인한다.

이유:

```text
`경기도 구리시`처럼 한 문자열에 광역과 시군구가 함께 들어오는 경우가 있다.
반대로 공고문에 잘못된 광역과 시군구가 함께 적힐 수도 있으므로 각각 매핑한 뒤 검증해야 한다.
```

예시:

```text
수집값=경기도 구리시
  -> 경기도=경기
  -> 구리시=경기,구리시
  -> 지역제한=경기,구리시

수집값=부산 중구
  -> 부산=부산
  -> 중구 후보 중 부산 중구로 좁혀짐
  -> 지역제한=부산,중구

수집값=경기 고성군
  -> 고성군은 강원/경남 후보
  -> 경기와 상하관계 불일치
  -> 검토필요
```

### 결정: 제외 지역 문구

`인천(옹진군 제외)`처럼 제외 조건이 있는 문구는 자동 처리할 수 있도록 구조를 준비한다.
현재는 사람이 확인한다.

이유:

```text
괄호 안 제외 조건은 단순 지역명 매핑보다 복잡하지만 자주 나오는 패턴이다.
파싱 규칙에서 기본지역과 제외지역을 분리할 수 있게 준비한다.
```

예시:

```text
수집값=인천(옹진군 제외)
  -> 기본지역=인천
  -> 제외지역=인천,옹진군
  -> 현재는 사람 확인
```

### 결정: 공사현장 일반 지역 매핑

공사현장이 일반 지역명으로 들어오면 지역제한과 같은 방식으로 매핑한다.
사업소형 값은 매핑하지 않고 원문을 유지한다.

이유:

```text
공사현장도 지역평가점수 계산에 쓰일 수 있으므로 일반 지역명은 표준 지역값으로 맞춰야 한다.
다만 사업소명은 지역디비와 겹치지 않으므로 원문을 보존한다.
```

예시:

```text
공사현장=파주/포천
  -> 공사현장=경기,파주시/경기,포천시

공사현장=남서울본부/강남지사
  -> 공사현장=남서울본부/강남지사
```

### 결정: 지역디비 보조열 삭제

지역디비의 뒤쪽 `Unnamed` 보조열은 운영 기준으로 쓰지 않고 삭제한다.

이유:

```text
P열은 R열의 시군구에서 `시/군`을 제거해 홈페이지용 값을 만들던 보조 수식이고,
R열은 하위구 row의 상위 시군구 보조값이다.
현재 필요한 값은 앞쪽 정식 컬럼에 이미 있으므로 설정 화면에 노출할 필요가 없다.
```

예시:

```text
P열 수식 결과=수원
H열 시군구-홈페이지용=수원
  -> P열 삭제
```

### 결정: 사업소·특수지역 별도 관리

배전사업소, 송전사업소, 폐광지역진흥지구는 일반 행정지역 마스터와 분리해 관리한다.
단, 화면 위치는 같은 `지역설정` 안에 둔다.

이유:

```text
행정지역은 주소지 표준화 기준이고,
배전/송전사업소는 한전 사업소 기준이며,
폐광지역진흥지구는 법정 특수 구역 기준이다.
성격과 컬럼이 다르므로 같은 테이블에 넣으면 유지보수가 어렵다.
```

예시:

```text
지역설정
  - 행정지역
  - 배전사업소
  - 송전사업소
  - 폐광지역진흥지구
```

### 결정: 사업소·특수지역 파일 구조

배전사업소와 송전사업소는 `id`, `지역본부`, `사업소명`, `인접 사업소` 구조로 관리한다.
폐광지역진흥지구는 `id`, `광역`, `시·군`, `상세 구역` 구조로 관리한다.

예시:

```text
배전사업소:
지역본부=서울
사업소명=강남지사
인접 사업소=서초지사, 강동송파지사

폐광지역진흥지구:
광역=강원
시·군=삼척시
상세 구역=도계읍
```

### 결정: 적격심사평가기준 매핑 원칙

적격심사평가기준 매핑은 A4 `적격심사세부기준`을 찾기 위한 전단계다.
이 분류가 틀리면 만점배수, 평가기준금액, 종목세부JSON까지 틀어지므로
확실한 조건으로만 자동 확정한다.

확정 규칙:

```text
금액 비교 기준=추정가격
금액 경계=미만
시행일=공고일 또는 입력일과 비교해 기준 후보 판단
복합 종목=현재는 주공종 기준만 사용
적격심사기준 표기=원문 보존
적격평가기준_세부=원발주처 + "_" + 적격심사기준
```

예시:

```text
원발주처=행자부
적격심사기준=별표-4
  -> 적격평가기준_세부=행자부_별표-4

추정가격=200,000,000
적격심사기준=2억미만
  -> 미만 조건이 아니므로 다음 구간 확인
```

### 결정: 적격심사기준 분류 구조는 참고자료를 그대로 복사하지 않는다

참고자료의 `master_evaluationCriteria.json`, `orgCriteria`, `rules_적격_skipKeywords.json` 같은 이름은
운영 예시로만 보고, 우리 시스템에서는 역할 기준으로 다시 나눈다.

이유:

```text
파일명과 현재 운영 방식은 바뀔 수 있지만,
우리 시스템에는 사람이 관리할 수 있는 룰관리 구조와 재전처리 흐름이 더 중요하다.
```

권장 구조:

```text
룰관리 > 적격심사기준
  - 기준대상/제외 입찰방식
  - 발주처 정규화
  - 1차 기준표
  - 2차 세부조건
  - 미등록 발주처
```

예시:

```text
참고명=master_evaluationCriteria.json
  -> 우리 구조=1차 기준표

참고명=orgCriteria
  -> 우리 구조=2차 세부조건
```

### 결정: 적격평가기준_세부 참고사항은 운영 원칙만 반영한다

기존 인계 문서는 잘 작동하는 참고 흐름이지만,
새 시스템 구조에는 함수명, 파일명, CSS명, 화면 경로를 그대로 고정하지 않는다.

반영할 운영 원칙:

```text
1. 1차 기준 매칭과 2차 세부조건 매칭을 분리한다.
2. 일반_기타_전문, 입력일, 추정가격 같은 선행값을 먼저 검증한다.
3. 같은 입력으로 여러 번 실행해도 같은 결과가 나오도록 멱등성을 지킨다.

5. 실패 사유와 사람이 고칠 방법을 기록한다.
```

예시:

```text
입찰방식=수의계약

입찰방식=적격심사
추정가격=빈값

```

### 결정: 확인/실적 설정의 검색용 메타

`공고확인`, `특수실적`, `특수실적_공통` 파일의 `종목`, `발주처`, `대분류`는 결과값을 검색하기 위한 메타로 본다.
공고 row의 종목/발주처/대분류와 자동 매칭하는 조건으로 쓰지 않는다.

이유:

```text
세 파일은 사람이 원하는 결과값을 찾고 관리하기 위한 설정 목록이다.
이 컬럼들을 공고 조건으로 해석하면 파서맨과 2차 매핑 조건이 뒤섞인다.
```

예시:

```text
특수실적 행:
종목=전기
발주처=한국철도공사
대분류=철도신호
결과값=철도신호설비_준공금액

위 종목/발주처/대분류는 사용자가 결과값을 찾는 필터다.
```

### 결정: 확인/실적 검색키워드 처리

검색키워드가 빈값인 행은 자동 파싱 대상이 아니라 사람이 직접 선택하거나 입력한다.
검색키워드 여러 줄은 OR 조건이며, 와일드카드를 사용할 수 있다.
같은 탭 안에서 결과값이 중복되면 데이터 오류로 본다.
한 공고에서 여러 결과값이 나오면 `/`로 연결한다.

예시:

```text
검색키워드:
전자시담
전자 시담

본문에 둘 중 하나가 있으면 결과값=전자시담을 반환한다.
```

### 결정: 공고확인과 특수실적의 2차 매핑 비교 방식

2차 매핑에서 `공고확인`은 전체 값이 정확히 일치해야 한다.
`특수실적`은 여러 값 중 조건값이 포함되어 있으면 일치로 본다.

예시:

```text
공고확인:
row=배전공가/전자시담
조건=배전공가
  -> 불일치

특수실적:
row=무정전_준공금액/철도신호설비_준공금액
조건=무정전_준공금액
  -> 일치
```

### 결정: 특수실적은 2차 매핑 전에 완료

`특수실적`은 적격심사기준 2차 매핑 전에 채워져야 한다.

이유:

```text
2차 매핑 조건이 특수실적을 참조할 수 있고,
특수실적에 따라 원발주처와 적격평가기준_세부가 달라질 수 있기 때문이다.
```

예시:

```text
특수실적=무정전_준공금액
  -> 2차 매핑 조건에서 무정전 기준으로 원발주처/적격평가기준_세부 덮어쓰기 가능
```

### 결정: 기존 곡괭이질 문서는 원칙만 참고

기존 `곡괭이질 버튼` 문서는 전체 처리 흐름과 실패 추적 방식을 참고한다.
하지만 기존 함수명, 파일명, 한 화면에 모든 책임이 모인 구조는 그대로 쓰지 않는다.

가져갈 원칙:

```text
대상 row 필터, 삭제키워드 필터, 실패 row 보존, 진행 UI, 파서 근거, 계산맨, 검증맨, 미등록 발주처 모음
```

버릴 구조:

```text
Raw 화면 함수가 본문 fetch, 파싱, 매핑, 계산, 종목분류, 적격매칭, 검증을 모두 직접 제어하는 구조
```

예시:

```text
새 구조에서는 `전처리실행기`가 각 모듈을 순서대로 호출하고,
파서맨/계산맨/적격매칭/검증맨은 독립적으로 테스트 가능해야 한다.
```

상세 기준:

```text
docs/25_곡괭이질_파서맨_계산맨_UIUX_통합분석_질문.md를 따른다.
```

### 결정: 수집은 서버공고만 원본으로 본다

수집 단계는 `서버공고(A1)`에서 모든 공고 컬럼 자료를 받는 단계다.
다른 발주처 홈페이지, A3 파싱용공고문, A4 적격심사세부기준은 수집 원본이 아니라 전처리 참조 데이터로 본다.
서버공고에서 모든 컬럼을 받되, `적격발주처` 값이 항상 채워진다고 보지 않는다.

이유:

```text
수집 원본을 여러 출처로 넓히면 서버공고 raw row와 공고문 파싱 후보가 섞인다.
새 구조에서는 서버공고 row를 원본으로 두고, A3/A4는 전처리에서 필요한 때만 참조한다.
```

예시:

```text
서버공고에서 `2601767-00` row의 모든 컬럼을 받는다.
이 row가 수집 원본이다.
`적격발주처` 컬럼이 빈값이면 수집 단계에서 억지로 채우지 않는다.
A3는 문서곡괭이가 빈값을 보완할 때 읽는 파싱용 본문이다.
A4는 적격평가기준_세부가 정해진 뒤 기준 row를 조회하는 참조 API다.
```

### 결정: 곡괭이질 3단계 구조

전처리 전체 파이프라인인 `곡괭이질`은 아래 3단계로 나눈다.

```text
1. 곡괭이전나라시
2. 문서곡괭이
3. 곡괭이후나라시
```

역할:

| 단계 | 역할 | 예시 |
|---|---|---|
| 곡괭이전나라시 | 서버공고 raw만 보고 가능한 전처리 | 삭제키워드, 괄호정리, 적격발주처변경, 투찰금액산식 |
| 문서곡괭이 | A3 HTML 공고문을 파싱해 값 후보 추출 | 종목, 공고확인, 특수실적 |
| 곡괭이후나라시 | 다른 컬럼을 참조해 계산 컬럼 산출 | 발주처코드 lookup, 예상투찰금액 계산 |

이유:

```text
문서 파싱 전에 가능한 raw 정리와,
문서 파싱 후에만 가능한 계산을 분리해야 책임이 섞이지 않는다.
```

예시:

```text
공고명 삭제기준은 공고문을 읽기 전에 서버공고 공사명만 보고 처리한다.
예상투찰금액은 파서맨이 A값/투찰율을 보완한 뒤 계산한다.
```

### 결정: 최종 엑셀 출력 컬럼

최종 엑셀은 일부 대표 컬럼만 출력하지 않고 모든 표준 컬럼을 출력한다.
컬럼 순서와 양식은 기존 사용 중인 최종 엑셀 양식에 가급적 맞춘다.

예시:

```text
공고번호, 공고명, 발주처명, 종목, 지역제한, 기초금액만 출력하는 방식이 아니다.
서버공고와 전처리에서 확정한 전체 표준 컬럼을 출력한다.
```

### 결정: 공고문 파일 파싱 기준

공고문 파일 자체(PDF/HWP/DOCX)를 직접 파싱하지 않는다.
문서곡괭이는 기본적으로 API에서 받아오는 A3 HTML을 사용한다.
A2 첨부 fallback은 구현하지 않는다.

예시:

```text
A3 파싱용공고문 HTML이 정상으로 있으면 A2 첨부파일의 HWP/PDF를 따로 변환하지 않는다.
```

### 결정: 적격심사 매칭 세부 원칙

2차 매핑에서 빈 조건은 비교에서 빠진다.
조건 설정이 올바르면 충돌은 발생할 수 없고, 충돌하면 조건세부 설정 오류로 본다.
조달청 기준명은 `별지`가 아니라 `별표` 기준으로 기록한다.

예시:

```text
2차 조건:
종목=빈값
공고확인=고난도

처리:
종목 조건은 보지 않고 공고확인만 비교한다.
```

### 결정: 적격 디버그 메타 컬럼명

적격심사 디버깅용 상태는 기존 `_v1_*` 이름 대신 사람이 읽기 쉬운 이름으로 둔다.

```text
적격_1차상태
적격_1차사유
적격_2차상태
적격_2차사유
적격_처치방법
```

예시:

```text
적격_1차상태=미매칭
적격_1차사유=추정가격 없음
적격_처치방법=추정가격 입력 필요
```

### 결정: 특수실적 보조 컬럼의 사용 범위

`특수실적`의 `상위실적`과 `특수실적_공통`의 `적격업체`는 사용자 검색용 메타다.
파서맨과 계산맨이 직접 사용하지 않는다.

예시:

```text
상위실적=a>b>c>d
  -> 사용자가 설정 화면에서 결과값을 찾기 위한 검색용

적격업체=true
  -> 사용자가 설정 화면에서 결과값을 찾기 위한 검색용
```

### 결정: 특수조건 입력 라우터

`특수조건`은 사람이 입력하는 원문 입력칸이며, 입력값은 남긴다.
계산맨/라우터가 필요한 값을 `공고확인` 같은 실제 참조 컬럼으로 복사한다.
`특수실적_공통`은 2차 매핑표가 직접 참조하지 않는다.
라우터는 입력 즉시 실행하지 않고 저장 또는 2차분류 버튼 시점에 실행한다.

예시:

```text
특수조건=기초금액발표전/사회적기업
  -> 특수조건 원문은 유지
  -> 공고확인=기초금액발표전
  -> 특수실적_공통=사회적기업
  -> 2차 매핑은 공고확인 같은 실제 참조 컬럼을 봄
```

### 결정: 파서맨 표시와 저장

파서맨 조건판단형태는 운영 화면에서 짧게 표시한다.
파서맨 후보 컬럼을 별도로 두 배로 늘리지 않고 대상 컬럼에 덮어쓴다.
근거는 상세모달에서 `rule_id`, `matched_keyword`, `source_text`로 확인할 수 있게 한다.

예시:

```text
조건판단형태 표시=3_1
결과 저장=공고확인 컬럼에 전자시담
근거=상세모달에서 rule_id/matched_keyword/source_text 확인
```

### 결정: 특수실적 누락 오류 표시

특수실적의 금액, 면적, 점수 값이 필요한데 비어 있으면 상세모달에서 빨간 오류로 표시한다.

예시:

```text
특수실적=공연장[3000M]_준공면적
준공면적 숫자 없음
  -> 상세모달에서 빨간 오류
```

### 결정: 남은 미정 항목 처리 방식

남은 질문은 지금 문답으로 모두 닫지 않는다.
구현 중 실제 데이터, 화면, 충돌 케이스가 나올 때 예시와 함께 결정한다.

예시:

```text
구현 중 단독평가종목=토목/전기이고 종목별 평가기준금액이 서로 다른 공고가 발견됨
  -> 그 케이스를 기준으로 종목별 A4 개별 호출 여부를 다시 결정
```
